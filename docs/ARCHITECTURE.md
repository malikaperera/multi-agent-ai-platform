# Roderick - Architecture

## Agent Hierarchy

```
You (the owner)
  └── Roderick - orchestrator + personal assistant (Telegram + dashboard)
        ├── Merlin    - research agent
        ├── Forge     - builder agent (two-stage approval + Sentinel gate)
        ├── Sentinel  - QA / validation agent (Forge gate)
        ├── Venture   - business opportunity agent
        ├── Zuko      - job search + application agent
        ├── Atlas     - personalised skill tutor (configurable domain)
        └── Operator  - business execution agent
```

---

## Process Model

All agents run as asyncio tasks inside a single Python process, sharing one SQLite database via WAL mode.

Each agent runs a polling loop:
1. Check the task queue for `pending` tasks addressed to it
2. Pick one up, mark it `in_progress`
3. Execute (LLM call, browser automation, file I/O, etc.)
4. Write a structured result back to the DB
5. Sleep `worker_poll_interval_seconds` and repeat

Agents can be extracted to separate containers without changing the DB contract - communication is entirely through the database, never direct function calls.

---

## Communication Model

All coordination flows through `data/roderick.db` (SQLite, WAL mode).

```
[Telegram / Dashboard]
         │
         ▼
 [Roderick orchestrator]
    classify + route
         │
    ┌────┼────────────────────────────────────┐
    ▼    ▼                                    ▼
[Agentic  [Task queue]               [Approval requests]
 loop]      │                                 │
            │     ┌──────────────────┐        │
            ├────►│ Merlin           │        │
            ├────►│ Forge ◄──────────┼────────┘
            ├────►│ Sentinel         │
            ├────►│ Venture          │
            ├────►│ Zuko             │
            ├────►│ Atlas            │
            └────►│ Operator         │
                  └──────────────────┘
                           │
              [Telegram + Dashboard: results to owner]
```

No agent calls another directly. Every handoff is a database row. This makes every interaction auditable and replayable.

## Database Schema

**`tasks`**: agent task queue. Full lifecycle:
```
pending → approved → in_progress → plan_ready → plan_approved → awaiting_validation → completed
                                                                                    → failed
        → rejected
```

**`agent_registry`**: registered agents with status, last heartbeat, and current task.

**`approval_requests`**: pending/resolved approvals linked to tasks.

**`agent_messages`**: inter-agent message queue (e.g. Roderick ↔ Zuko, reflection messages).

**`reminders`**: time-based personal items with optional due date and category.

**`events`**: audit log of all significant agent actions.

**`agent_errors`**: error log per agent, used for watchdog and diagnostics.

**`behaviors`**: runtime-configurable agent behavior policies (key/value).

## Forge Two-Stage Approval Flow

```
User: "build X"
  → Roderick creates Task(to_agent='forge', status='pending')
  → ApprovalRequest(type='task_approval') sent via Telegram inline keyboard
  → Owner clicks APPROVE
  → Task.status = 'approved'
  → Forge plans → Task.status = 'plan_ready'
  → ApprovalRequest(type='plan_approval') sent via keyboard
  → Owner clicks APPROVE
  → Task.status = 'plan_approved'
  → Forge builds → Task.status = 'awaiting_validation'
  → Sentinel validates → Task.status = 'completed'
```

`system_improvement` tasks (generated by agent reflection) can be configured to skip the first approval gate via `forge.auto_approve_improvements: true`. The Sentinel gate is always enforced.

Forge **never** creates files before `plan_approved`.

## LLM Adapter

`shared/llm/adapter.py` defines the `LLMAdapter` protocol:
- `complete(messages, system, name)`: single-shot response
- Implementations: `AnthropicProvider`, `OllamaProvider`

Provider is selected by `llm.provider` in `config/roderick.json`:
- `"anthropic"`: Anthropic API (Claude models)
- `"ollama"`: local Ollama instance (any model)

Each agent can specify its own model via config (e.g. Forge uses a separate planner and coder model). The `LLMFactory` in `shared/llm/factory.py` builds the right provider from config.

## Scheduler

APScheduler `AsyncIOScheduler` sharing the main asyncio event loop.

Configured schedules (all times in `scheduler.timezone`, default UTC):

| Job | Trigger | Default time |
|---|---|---|
| `daily_briefing` | cron | 08:00 |
| `atlas_daily_lesson` | cron | 09:00 |
| `weekly_planning` | cron (Sunday) | 19:00 |
| `approval_digest` | cron (multi-slot) | 09:00, 14:00, 20:00 |
| `reminder_check` | interval | every 15 min |
| `agent_watchdog` | interval | every 10 min |
| `merlin_proactive_research` | interval | every 2 hours |
| `zuko_proactive_scan` | interval | every 3 hours |
| `venture_proactive_research` | interval | configurable |

All schedules are tuneable in `config/roderick.json`.

## Self-Improvement Loop

After each completed task, agents run a reflection step:
1. The agent's LLM reviews what it learned
2. If a concrete improvement is identified, it enqueues a `system_improvement` task for Forge
3. With `auto_approve_improvements: true`, Forge picks it up without manual approval
4. Sentinel validates the build before promotion

This creates a self-improving loop where all changes still pass through Sentinel.

## Owner Memory

`shared/memory/founder.py`: `OwnerMemory` class.

Reads from `memory/*.md` files at runtime:
- `owner_profile.md`: role, goals, background
- `current_focus.md`: what you are working on right now
- `active_projects.md`: projects and their status
- `capital_state.md`: financial context for Venture decisions
- `job_market.md`: job market data used by Atlas and Zuko
- `opportunity_log.md`: opportunity history for Venture

All memory files are plain markdown - edit them directly to update agent context.

## Presence System

Roderick adjusts delivery mode based on `presence.default_mode` in config:

| Mode | Behaviour |
|---|---|
| `at_pc` | Dashboard-first. Telegram for critical alerts and approvals only. |
| `away` | Telegram-first. All messages and updates via Telegram. |
| `dnd` | Critical alerts only. No briefings, no routine messages. |
| `focus` | Reduced noise. Non-critical items batched. |

Presence can be changed via Telegram or the dashboard.

## Adding a New Agent

1. Add an entry to `config/agents.json`
2. Create `apps/<agent_name>/agent.py` with a `run()` async polling loop
3. Register the task type in `config/roderick.json` classification categories
4. Wire `asyncio.create_task(agent.run())` in `apps/roderick/main.py`
5. That's it - the task queue, approval flow, heartbeats, and watchdog work automatically
