# Multi-Agent AI Platform

A self-hosted multi-agent AI platform controlled via Telegram. Eight specialist agents collaborate autonomously, doing research, writing and shipping code, searching for jobs, and identifying business opportunities, while routing all changes through a validation gate and a two-stage human approval flow.

---

## Architecture

```
You (via Telegram)
  └── Roderick (orchestrator + personal assistant)
        ├── Merlin   (research agent)
        ├── Forge    (builder agent - two-stage approval + Sentinel gate)
        ├── Sentinel (QA / validation agent)
        ├── Venture  (business opportunity agent)
        ├── Zuko     (job search agent)
        ├── Atlas    (skill tutor - configurable domain)
        └── Operator (business execution agent)
```

All coordination flows through a shared SQLite database (WAL mode). No agent calls another directly. They communicate through task queue rows and event rows, which makes every handoff auditable and replayable.

## Agents

### Roderick
The central coordinator. Receives all Telegram messages, classifies intent, routes to the right agent, and returns results. Runs the daily briefing, manages reminders, and has direct visibility into every agent's queue.

### Merlin
Handles deep research tasks. Given a question, Merlin searches, synthesises findings into structured JSON, and surfaces options with confidence scores and recommended next steps.

### Forge
Writes and ships code improvements. Every task requires two approval steps (plan approval, then implementation approval) before Forge touches the codebase. All changes are validated by Sentinel before promotion.

`system_improvement` tasks (generated by agents reflecting on their own work) can be configured to auto-approve, enabling a self-improving loop without manual intervention.

### Sentinel
Acts as a gate before any Forge implementation is marked complete. Checks correctness, security, and regression risk. Forge cannot promote work that Sentinel rejects.

### Venture
Continuously scans configured domains for market gaps, pricing signals, and revenue opportunities. Returns structured execution intelligence: revenue model, time to first dollar, capital required, automation potential.

### Zuko
Autonomous job search pipeline. Fetches listings from configured job boards, scores them against a candidate profile (title fit, keyword alignment, seniority guardrails), generates tailored cover letters, and drives browser-based applications with owner approval. Agency-posted jobs are detected and the cover letter is addressed to the client company, not the recruiter. All activity is tracked in the shared database.

### Atlas
A personalised learning coach. Delivers structured lessons, tracks skill progression across configurable states (`unknown > introduced > learning > practiced > project_used > interview_ready`), and uses the owner's real system as the learning environment. Domain and curriculum topics are set in `config/roderick.json` and can be anything: cloud engineering, data science, a programming language, etc.

### Operator
Sits downstream of Venture. Takes approved initiatives and turns them into concrete, sequenced action plans. Enforces strict approval gates before any spend, external outreach, or identity decisions. Coordinates Forge, Merlin, Sentinel, and Venture for execution work on configured ventures.

## Agent Approval Pipeline

```
pending > approved > in_progress > plan_ready > plan_approved > awaiting_validation > completed
                                                                                   > failed
```

- **pending > approved**: owner approves the task (or auto-approved for system improvements)
- **plan_ready > plan_approved**: owner approves Forge's implementation plan
- **awaiting_validation > completed**: Sentinel validates the build

All other tasks (non-system-improvement) require explicit owner approval at both gates.

---

## Self-Improvement Loop

Agents run a reflection step after each completed task. If a reflection identifies a concrete system improvement, it enqueues a `system_improvement` task for Forge. With `auto_approve_improvements: true` in config, Forge picks these up without manual approval. The system improves itself while all changes still pass through Sentinel.

## Quickstart

### Prerequisites
- Docker Desktop
- Telegram bot token ([BotFather](https://t.me/BotFather))
- Anthropic API key (or local Ollama running at port 11434)

### Setup

```bash
git clone https://github.com/malikaperera/multi-agent-ai-platform.git
cd multi-agent-ai-platform

cp .env.example .env
# Fill in: TELEGRAM_BOT_TOKEN, ANTHROPIC_API_KEY, AUTHORIZED_CHAT_ID

# Optional: configure your ventures in config/roderick.json
# Optional: place your CV at ./cv/your_cv.pdf for Zuko job applications

docker compose up -d
```

### Configuration

| File | Purpose |
|---|---|
| `.env` | Secrets (bot token, API keys, auth IDs) |
| `config/roderick.json` | Agent behaviour, thresholds, prompts, schedules |
| `config/agents.json` | Agent registry (names, autonomy levels, capabilities) |
| `apps/zuko/config/job_targeting.json` | Job search keywords, role targets, exclusions |
| `apps/zuko/config/candidate_profile.json` | CV summary and cover letter style (create from `.example.json`) |

## Design Decisions

- **Single SQLite database**: all agents share one DB via WAL mode. No message broker, no service mesh. Simple, auditable, and runs on minimal hardware.
- **No direct agent-to-agent calls**: all communication is through the task queue. Every handoff is a database row.
- **Two-stage approval for Forge**: plan approval and implementation approval are separate steps. You can see what Forge intends before it writes a line of code.
- **Sentinel as hard gate**: Forge cannot self-promote. All builds require external validation.
- **Local LLM fallback**: Ollama support means the system runs air-gapped or cost-free with models like Qwen or Llama.
